# LifeQuest 笔记本文件管理器设计

## 背景

当前笔记功能更接近“笔记本 -> 文件夹 -> 笔记”的二级结构：

- 笔记创建必须传入 `folder_id`，根目录不能直接创建笔记。
- 数据模型虽然有 `Folder.parent_id`，但前端和 API 没有完整支持多级目录操作。
- 笔记文件路径由 `folder_id + sanitize(title).md` 生成，同一目录同名笔记存在内容覆盖风险。
- 用户预期是“笔记本就是一块文件夹空间”：打开笔记本后像文件管理器一样操作根目录、多级目录、笔记文件。

本设计只覆盖笔记本/目录/笔记的文件管理行为，不扩展附件、协作、版本历史、全文检索等非必要能力。

## 目标

- 每个笔记本表现为一个独立的文件夹空间。
- 根目录可以直接创建笔记和文件夹。
- 任意目录可以创建子目录和笔记，支持多级嵌套。
- 同一目录下禁止同名条目，避免路径冲突和内容覆盖。
- 支持创建、读取、重命名、移动、删除目录和笔记。
- 后端 API、数据库、磁盘 Markdown 文件路径保持一致。
- 前端提供类似文件管理器的树形导航和当前目录内容列表。

## 非目标

- 不要求真实操作系统文件夹作为唯一数据源。
- 不实现多人协作、权限共享、回收站、版本历史。
- 不在本次重构中实现附件上传。
- 不要求前端新增自动化测试框架；前端以构建和手工验收为主。

## 推荐方案

采用统一文件节点模型。

保留 `notebooks` 作为笔记本元数据表。新增或迁移到一个统一的节点表，用同一张表表示文件夹和笔记：

- `folder` 节点表示目录。
- `note` 节点表示 Markdown 笔记。
- `parent_id = null` 表示位于笔记本根目录。

这样后端只需要维护一棵树，前端也只需要渲染一棵树，根目录、子目录、移动、重命名和同名校验都可以走同一套逻辑。

不推荐继续只做小修补。让 `Note.folder_id` 可为空虽然能快速支持根目录笔记，但会继续保留 `folders` 和 `notes` 两套不同层级模型，后续做移动、重命名、树形 UI、路径同步时复杂度会持续增加。

## 数据模型

### notebooks

沿用现有表：

- `id`
- `user_id`
- `name`
- `description`
- `icon`
- `created_at`

### note_nodes

新增统一节点表：

- `id`: UUID，主键
- `notebook_id`: UUID，外键
- `parent_id`: UUID，可空，自引用；为空表示根目录条目
- `type`: `folder` 或 `note`
- `name`: 显示名；文件夹不带扩展名，笔记显示为标题
- `normalized_name`: 规范化名称，用于同目录唯一校验
- `path`: 物化路径，例如 `/项目A/需求/验收.md`
- `content_path`: Markdown 文件相对路径；仅 `note` 节点有值
- `summary`: 笔记摘要；仅 `note` 节点有值
- `tags`: JSON 字符串；仅 `note` 节点有值
- `is_pinned`: 是否置顶；仅 `note` 节点有值
- `word_count`: 字数；仅 `note` 节点有值
- `created_at`
- `updated_at`

约束：

- 同一 `notebook_id + parent_id` 下，`normalized_name` 唯一。
- `type = folder` 时不允许有 `content_path`。
- `type = note` 时必须有 `content_path`。
- 禁止形成循环父子关系。

### 同名规则

同一目录下不允许出现规范化后同名的任何条目。

规范化规则：

- 去除首尾空白。
- 不允许空名称。
- 不允许 `/`、`\`、控制字符和 Windows 保留文件名。
- 比较时大小写不敏感。
- 笔记的 `.md` 扩展名不参与用户输入；用户输入标题 `周报`，系统路径为 `周报.md`。

示例：

- `/项目A/周报.md` 已存在时，不能再创建标题为 `周报` 的笔记。
- `/项目A/需求` 文件夹已存在时，不能再创建标题为 `需求` 的笔记。
- `/项目A/需求.md` 笔记已存在时，不能再创建名为 `需求` 的文件夹。

## 文件存储

Markdown 文件继续存放在后端 `notes_data` 下，但路径必须由节点树决定，而不是由 `folder_id/title` 临时拼接。

推荐结构：

```text
backend/notes_data/
  <user_id>/
    <notebook_id>/
      项目A/
        会议记录.md
        需求/
          验收.md
      Inbox.md
```

数据库是权限、索引和树结构的主数据源。磁盘 Markdown 文件是笔记正文的存储载体。

路径安全要求：

- 所有磁盘路径必须在 `notes_data` 根目录内解析。
- 文件名必须经过统一 sanitizer。
- 重命名和移动时同步移动 Markdown 文件。
- 如果文件移动失败，数据库事务必须回滚或返回失败，不能留下数据库路径和磁盘文件不一致。

## API 设计

所有接口必须校验当前用户拥有对应笔记本或节点。

### 笔记本

- `GET /api/notes/notebooks`
- `POST /api/notes/notebooks`
- `GET /api/notes/notebooks/{notebook_id}`
- `PUT /api/notes/notebooks/{notebook_id}`
- `DELETE /api/notes/notebooks/{notebook_id}`

### 节点树

- `GET /api/notes/notebooks/{notebook_id}/tree`
  - 返回整个节点树，供左侧树形导航使用。
- `GET /api/notes/notebooks/{notebook_id}/children?parent_id=...`
  - `parent_id` 为空时返回根目录条目。
  - 返回当前目录下的文件夹和笔记列表。
- `POST /api/notes/notebooks/{notebook_id}/folders`
  - 创建文件夹。
  - 入参：`parent_id`、`name`。
- `POST /api/notes/notebooks/{notebook_id}/notes`
  - 创建笔记。
  - 入参：`parent_id`、`title`、`content`、`summary`、`tags`。
- `PATCH /api/notes/nodes/{node_id}`
  - 重命名或移动节点。
  - 入参：`name`、`parent_id`。
- `DELETE /api/notes/nodes/{node_id}`
  - 删除文件夹或笔记。
  - 删除文件夹时递归删除子节点和对应 Markdown 文件。

### 笔记正文

- `GET /api/notes/{note_id}`
  - 返回笔记元数据和 Markdown 正文。
- `PUT /api/notes/{note_id}`
  - 更新标题、正文、摘要、标签、置顶状态。
  - 标题变更等同于重命名，需要执行同目录唯一校验和文件移动。

### 错误语义

- `400`: 名称非法、移动到自身子级、请求类型不匹配。
- `401`: 未认证。
- `403`: 当前用户无权限。
- `404`: 笔记本或节点不存在。
- `409`: 同一目录存在同名条目。
- `500`: 文件系统或数据库一致性错误。

## 前端设计

### 页面结构

笔记模块保留现有主入口 `/notes`，笔记本详情页改为文件管理器布局：

- 左侧：笔记本树，展示根目录、文件夹和笔记。
- 中间：当前目录内容列表，文件夹和笔记混排。
- 右侧或独立页面：Markdown 编辑器。

### 核心交互

- 在根目录创建文件夹。
- 在根目录创建笔记。
- 在任意目录创建子目录。
- 在任意目录创建笔记。
- 点击文件夹进入目录。
- 点击笔记进入编辑器。
- 重命名文件夹或笔记。
- 移动文件夹或笔记到其他目录。
- 删除文件夹或笔记。
- 同名冲突时显示明确错误：`当前目录已存在同名条目`。

### 路由建议

- `/notes`: 笔记本列表。
- `/notes/:notebookId`: 打开笔记本根目录。
- `/notes/:notebookId/folder/:folderId`: 打开指定目录。
- `/notes/:notebookId/edit/:noteId`: 编辑指定笔记。
- `/notes/:notebookId/new`: 在根目录新建笔记。
- `/notes/:notebookId/folder/:folderId/new`: 在指定目录新建笔记。

## 迁移策略

现有数据迁移到 `note_nodes`：

1. 为每个现有 `Folder` 创建 `folder` 节点。
2. 按 `Folder.parent_id` 建立父子关系。
3. 为每个现有 `Note` 创建 `note` 节点，父节点指向原 `folder_id` 对应节点。
4. 按新路径规则生成 `path` 和 `content_path`。
5. 移动旧 Markdown 文件到新路径。
6. 如果遇到同目录同名笔记，迁移必须失败并报告冲突清单，不允许静默覆盖。

SQLite 当前没有正式迁移体系。实现时可以先写一次性迁移函数，但上线前建议引入 Alembic 或明确的版本化迁移脚本，避免启动时隐式改表造成测试和生产不可控。

## 测试与验收

### 后端测试

必须覆盖：

- 根目录创建笔记成功。
- 根目录创建文件夹成功。
- 子目录创建笔记成功。
- 多级子目录创建成功。
- 同一目录创建同名笔记返回 `409`。
- 同一目录创建同名文件夹返回 `409`。
- 同一目录创建同名文件夹和笔记返回 `409`。
- 不同目录允许相同名称。
- 重命名成同目录已有名称返回 `409`。
- 移动到目标目录已有同名条目返回 `409`。
- 删除目录会递归删除子节点和 Markdown 文件。
- 普通用户不能访问其他用户的笔记本和节点。
- 文件路径不能逃逸 `notes_data`。

### 前端验收

必须手工验证：

- 打开笔记本后能看到根目录内容。
- 根目录能创建笔记。
- 根目录能创建文件夹。
- 文件夹内能创建子文件夹。
- 文件夹内能创建笔记。
- 树形导航能展示多级目录。
- 点击笔记能进入编辑器并保存正文。
- 同名冲突有明确提示，不会覆盖旧内容。
- `npm run build` 通过。

### 上线门槛

笔记功能达到以下条件后才建议上线：

- 后端新增测试全部通过。
- 现有后端测试全部通过。
- 前端生产构建通过。
- 至少完成一轮手工验收，覆盖根目录、多级目录、同名冲突、编辑保存、删除。
- 迁移脚本在测试数据库上跑通，且同名冲突可被报告。

## 开放风险

- 当前仓库已有较多未提交改动，实施前需要确认哪些是开发修复，避免误回滚。
- 现有前端部分中文内容疑似编码损坏，重构笔记页面时应同步修复相关文案文件为 UTF-8。
- 若继续使用启动时自动补列的方式做数据库变更，测试隔离和生产迁移风险仍然存在。
